## 71. STATE MANAGEMENT (Riverpod / Bloc)

- One provider per section → independent rebuilds, no full-screen jank.
- `dashboardControllerProvider` orchestrates; child providers stream slices.
- **Optimistic UI:** favorite/pin/delete update instantly, reconcile on server ack.
- **Cache-first:** hydrate from local (Hive/Isar) → render immediately → refresh in background → diff-patch the UI.

```
Providers
 ├── greetingProvider          (time + usage + weather)
 ├── heroCarouselProvider      (7 slides, auto 5s)
 ├── quickActionsProvider      (personalized order)
 ├── resumeSessionProvider     (Smart Continue)
 ├── topToolsProvider          (recommendation engine §28)
 ├── categoriesProvider
 ├── recentWorkProvider        (realtime via Socket.IO)
 ├── analyticsProvider
 └── syncStatusProvider        (§29 live status bar)
```

## 72. AI PERSONALIZATION ENGINE (Architecture)

Powers §23 greeting, §28 recommendations, §44 personalization, §50 dynamic dashboard.

```
Signals ──► Feature Store ──► Ranking Model ──► Layout Resolver ──► UI
(taps, dwell,          (on-device +        (lightweight       (section order,
 favorites, time,       Supabase edge)      gradient-boost      tool order,
 profession, tool                            or heuristic)      greeting, hero)
 sequences)
```

- **Privacy-first:** primary ranking runs **on-device**; only anonymized aggregates sync to Supabase.
- **Cold start:** onboarding profession chip seeds initial tool order until real signals accumulate.
- **Explainability:** each recommendation carries a `reason` ("You used this 4× this week") shown on long-press.
- **Fallback:** if the model is unavailable, deterministic popularity ranking is used — dashboard never renders empty.

## 73. REALTIME & SYNC

| Channel | Transport | Drives |
|---|---|---|
| Job progress | Socket.IO | Recent Work progress bars, Smart Continue |
| Cloud sync | Supabase Realtime | §29 status strip, §36 live cloud |
| Notifications | Push (FCM) + in-app | §37 / §59 notification hub |
| Presence | Supabase Realtime | §66 cross-device live presence |

- Reconnect with exponential backoff; show a subtle "Reconnecting…" chip, never a blocking spinner.
- All realtime writes are **idempotent** (keyed by `jobId`) so retries never double-apply.

---

## 74. OFFLINE-FIRST BEHAVIOR

- Dashboard shell, last snapshot, favorites, and offline-capable tools render with **zero network**.
- Tool cards show an **Offline badge** (§56) when usable without connection (client-side tools: QR, hash, text, image resize/compress).
- Queued jobs created offline flush automatically on reconnect (write-behind queue in Isar).
- Analytics events buffer locally and batch-upload when online (§10 taxonomy preserved).

## 78. SECURITY (Mobile Specifics)

- Local storage encrypted (Isar/Hive + platform keystore/keychain).
- Session tokens in secure storage only; never in shared prefs.
- Certificate/SSL pinning on all API + realtime endpoints.
- Biometric gate (`local_auth`) optional for app open and the Biometric Vault (§65).
- Screenshot/recording protection (`FLAG_SECURE`) on sensitive surfaces (billing, vault, AI chat with private files).
- Signed, expiring URLs for every file thumbnail — no public R2 links in the UI.

## 80. API CONTRACTS (Dashboard)

Standard envelope (matches web spec §16): `{ success, data, error, meta }`.

| Route | Method | Returns |
|---|---|---|
| `/api/dashboard` | GET | Full `DashboardState` snapshot (single hydration call) |
| `/api/dashboard/greeting` | GET | `GreetingData` (time/weather/usage aware) |
| `/api/tools/recommended` | GET | Ranked `ToolCard[]` for this user |
| `/api/jobs/recent` | GET | `RecentWork[]` (paginated) |
| `/api/analytics/summary` | GET | `AnalyticsSummary` (week/month) |
| `/api/workspaces` | GET/POST | List / create workspaces (§48) |
| `/api/dashboard/layout` | PUT | Persist customized widget/section order (§33/§50) |

Design rule: the home screen boots from **one** `/api/dashboard` call (fast first paint), then sub-routes refresh individual sections in the background.
