# ⚠ Breaking Changes

Breaking changes should only be introduced in major releases.

Examples include

- Major API redesign
- Database schema migration
- Authentication architecture changes
- Core module restructuring
- Deprecated platform support
- Package replacement requiring migration

Every breaking change must include:

- Detailed explanation
- Reason for change
- Migration steps
- Compatibility notes
- Rollback strategy (if applicable)

# 🔄 Migration Policy

Whenever a release introduces incompatible changes, a migration guide must be provided.

Migration documentation should include:

- Prerequisites
- Version compatibility
- Upgrade procedure
- Configuration changes
- Data migration
- Validation steps
- Troubleshooting
- Rollback instructions

Migration guides should be stored with the corresponding release documentation.
